01
⚡ Most common blocker
Gateway webhook isn't receiving messages
Your agent is running, but nothing arrives. Messages from Telegram disappear into the void.
✓
Done
↓ Work through these steps
✓
Confirm the gateway process is actually running. Run the command below — if it returns nothing, the gateway isn't started.
openclaw gateway status # Expected: "Gateway is running on port ..." # If not running: openclaw gateway start
✓
Check that your gateway port is reachable from the internet. The port number is shown in your OpenClaw dashboard under Settings → Gateway. If you're on a home network, your router is blocking it. You need either (a) port forwarding rules in your router admin panel, or (b) a tunnel like ngrok.
# Option B — quickest to test with ngrok: # Replace YOUR_PORT with the port shown in your OpenClaw dashboard ngrok http YOUR_PORT # Gives you a public URL like: https://abc123.ngrok-free.app
✓
Update your webhook URL in BotFather (or whichever channel you're using). The URL must be the public-facing URL pointing to your gateway, not localhost.
# Telegram: set via BotFather → /setwebhook # Or via Telegram API directly: curl "https://api.telegram.org/botYOUR_BOT_TOKEN/setWebhook?url=https://YOUR_PUBLIC_URL/webhook" # YOUR_BOT_TOKEN = the token BotFather gave you (e.g. 123456:ABC-DEF...) # YOUR_PUBLIC_URL = the ngrok or VPS URL from the previous step
✓
Verify the tunnel is actually passing traffic. Send a test POST directly to your webhook URL.
curl -X POST https://YOUR_PUBLIC_URL/webhook \ -H "Content-Type: application/json" \ -d '{"test": true}' # Any response = tunnel is live. Silence = port still blocked. # YOUR_PUBLIC_URL = the ngrok / VPS address from step 2
Permanent fix: ngrok works for testing, but the URL changes every restart. For a stable setup, either use a VPS with a fixed IP + nginx reverse proxy, or use Tailscale to expose the port without opening your router.

02
😵 Settings overload
openclaw.json has hundreds of fields — don't know what to touch
You open the config file and see fields you've never heard of. It's not clear what's required, what's dangerous to change, and what order things go in.
✓
Done
↓ Work through these steps
✓
For a first working setup, only 4 paths matter. Every other field can stay at its default value until you know what you're doing.
{ "agents": { "defaults": { "model": { "primary": "claude-sonnet-4-6" } // which model to use } }, "gateway": { "port": YOUR_PORT // shown in Settings → Gateway }, "channels": { "telegram": { "botToken": "YOUR_BOT_TOKEN" } // from BotFather }, "models": { "providers": { "anthropic": { "apiKey": "YOUR_ANTHROPIC_KEY" } } } } // Orange values = fields you need to fill in yourself // Everything else can stay as-is for now
✓
All keys live in the single openclaw.json file — there is no separate keys file. Model API keys and channel tokens are set in different sections of the same file.
// Model API key (e.g. Anthropic): "models": { "providers": { "anthropic": { "apiKey": "sk-ant-YOUR_ANTHROPIC_KEY" } } } // Telegram bot token: "channels": { "telegram": { "botToken": "YOUR_BOT_TOKEN_FROM_BOTFATHER" } } // Get Anthropic key: console.anthropic.com → API Keys // Get Telegram token: Telegram → search @BotFather → /newbot
✓
After editing config, always restart the gateway. OpenClaw does not hot-reload config changes in all situations. Run these two commands one at a time — wait for the first to finish before running the second.
# ① Run first — stops the gateway: openclaw gateway stop # ② Run after ① completes — starts the gateway: openclaw gateway start # ③ Check that it started correctly: openclaw gateway status

03
🔌 Skills not working
Skill is enabled in settings, agent still can't use it
You toggled the skill on, restarted, but the agent either ignores it or throws errors when it tries to use it.
✓
Done
↓ Work through these steps
✓
Enable skills one at a time. Don't batch-enable 10 skills at once. Add one, restart, confirm it works in the logs, then add the next. Batch enabling makes it impossible to tell which one broke something.
✓
Each skill has its own auth requirements beyond the toggle. Check the skill's configuration panel in the OpenClaw dashboard. Most require a separate API key or OAuth token — the toggle alone does nothing without credentials.
# Skills with separate auth requirements (common ones): gmail → Google OAuth2 token gcal → Google OAuth2 token slack → Slack bot token + workspace ID github → GitHub personal access token google-drive → Google OAuth2 token
✓
Run the gateway in verbose mode to check for permission errors after enabling a skill.
openclaw gateway stop openclaw gateway run --verbose # Look for: "permission denied", "invalid token", "scope insufficient" # These tell you exactly which credential is wrong or missing. # Ctrl+C to stop, then: openclaw gateway start

The agent has been running a few days. You open the Anthropic console and see $30, $60, or more. No idea where it went.
✓
Done
↓ Work through these steps
✓
Set a hard monthly spend limit now, before anything else. Go to console.anthropic.com → Settings → Limits. Set a hard cap. The API will stop processing requests when you hit it — no surprises.
# Recommended starting limits: Soft limit: $10 → you get an email warning Hard limit: $20 → requests stop dead here # A well-configured single agent doing 10–20 turns/day costs ~$5–15/month. # If you're burning $30 in 3 days, there's a loop or a massive prompt.
✓
Check your system prompt token count. A 10,000-token system prompt sent on every turn will drain budget fast. Count tokens before you deploy.
# Quick estimate: 1 token ≈ 0.75 words # A 2,000-word system prompt ≈ ~2,700 tokens # At $3/M input tokens (Sonnet): 2,700 × 20 turns/day × 30 days # = ~$4.86/month just for the system prompt # Add tool calls, context, output: easily 3–5× that
✓
Keep your system prompt short — and instruct your agent to do the same in its replies. This is the highest-leverage cost lever you control directly. Here's what the math looks like at 20 turns/day:
# System prompt: 10,000 tokens vs 2,000 tokens # Difference: 8,000 tokens × 600 turns/month # At $3/M input tokens (Sonnet): # → $14.40/month saved — just from trimming the prompt # # Add this to your agent's soul/constitution document:
# Paste into your agent's soul or constitution file: "Keep all responses concise. Do not repeat context from previous turns — summarize instead of quoting. If a reply exceeds 400 words, ask yourself: can this be shorter? Target: under 300 words per reply unless the task requires more."
✓
Check for runaway loops in your gateway logs. Silent loops (agent retrying a failed task indefinitely) are the most common cause of unexpected bills.
# Stop the service and run in foreground to watch for loops: openclaw gateway stop openclaw gateway run --verbose # Watch for the same tool call repeating rapidly → loop detected # Ctrl+C to stop, then: openclaw gateway start # Add a max_turns limit to your agent config to cap runaway turns

You've read conflicting advice. Some say VPS is cheaper, some say Mac Mini is more reliable. You haven't started because you can't decide.
✓
Done
↓ Work through these steps
✓
Use this decision rule and stop second-guessing it:
Already own a Mac? → Mac Mini (zero infra cost) Want to start without hardware? → VPS ($5–10/month) Need to share with the team? → VPS (easier access control) Want the simplest possible debug? → Mac Mini (you have a screen)
✓
If going VPS: understand what you're actually getting over a home machine. VPS has real advantages worth knowing before you decide.
Security Pre-hardened datacenter infrastructure. Your home IP and personal network stay completely unexposed. Agent traffic never passes through your home router. Privacy Agent logs stay on a machine physically separate from your personal data. Easier to wipe and rebuild without touching anything personal. Scale Upgrade RAM/CPU in a few clicks — no hardware purchase, no downtime. Add a second agent on a separate VPS without buying new machines. Team access Multiple people can SSH in with separate keys — no "it's on my Mac" problems. # For minimum specs, check: docs.openclaw.ai → Installation # Common providers: Hetzner (EU), DigitalOcean, AWS Lightsail
✓
If buying a Mac Mini: pick your RAM based on how many projects you're planning.
1 project (getting started): 16GB RAM — sufficient Multiple projects / scaling: 32GB RAM — start here if you already have several automations in mind Storage: not a bottleneck for beginners — base SSD is fine Model: any current Mac Mini (M2 / M4) handles OpenClaw without issues # Rule of thumb: if you're already planning more than one agent # or pipeline, the 32GB upgrade is cheaper than buying again later.
✓
If going Mac Mini: set up Tailscale for a stable public address without touching your router.
# Tailscale gives your Mac a stable IP accessible from anywhere, # no port forwarding needed. Free tier covers personal setups. # Run each command separately, one at a time: # ① Install Tailscale: brew install tailscale # ② Start the daemon: sudo tailscaled # ③ Connect to your Tailscale account: tailscale up
Reality check: The hardware is not the hard part. Both options work fine. The config is where people get stuck — and that's the same regardless of where the agent runs.

06
🧠 Memory wiped
Agent's memory resets after a few days
It was remembering things fine. Then it acts like it never spoke to you. All context is gone.
✓
Done
↓ Work through these steps
✓
Find where your memory files are actually being written. OpenClaw has no separate memoryPath setting — memory lives inside the workspace directory set at agents.defaults.workspace.
# Check the workspace path in your openclaw.json: "agents": { "defaults": { "workspace": "/your/workspace/path" } } # Default if not set: ~/.openclaw/workspace # Verify memory files exist inside the workspace: ls -la ~/.openclaw/workspace/ # If the workspace directory is empty → agent has no persisted memory
✓
Always use an absolute path for the workspace. Relative paths resolve differently depending on where the gateway process was started from.
// ❌ Breaks after restart (relative path): "agents": { "defaults": { "workspace": "./workspace" } } // ✅ Safe (absolute path): "agents": { "defaults": { "workspace": "/Users/YOUR_USERNAME/.openclaw/workspace" } } // Mac "agents": { "defaults": { "workspace": "/home/YOUR_USERNAME/.openclaw/workspace" } } // Linux/VPS // YOUR_USERNAME = your system username (run: whoami)
✓
Set up a daily backup of the memory directory. Memory built over weeks is irreplaceable. One accidental rm or failed update wipes it.
# Add to crontab (crontab -e): 0 3 * * * cp -r ~/.openclaw/memory ~/.openclaw/memory_backup_$(date +%Y%m%d) # Runs daily at 3am, keeps dated backups # Clean old backups weekly to save space

07
👻 Silent failures
Agent stops responding or loops with no error
Messages go unanswered. Or the agent keeps doing the same thing over and over. No alert, no log, no indication anything is wrong.
✓
Done
↓ Work through these steps
✓
Run the gateway in verbose foreground mode to surface swallowed errors. There is no log viewer command — stop the service and run it in the foreground instead.
# ① Stop the background service: openclaw gateway stop # ② Run in foreground with verbose output (Ctrl+C to stop): openclaw gateway run --verbose # All errors, tool calls, and turn events print here in real time. # ③ When done debugging, restart as background service: openclaw gateway start
✓
Set up a daily heartbeat check. Schedule a cron that pings the agent and delivers the reply to your Telegram. If no reply within 30 minutes, the agent is stuck.
# crontab -e — runs every morning at 8am: 0 8 * * * openclaw agent --agent ops --message "Daily status check" --deliver # If you don't get a Telegram message by 8:30 → something is broken
✓
Add explicit stop conditions to your agent's instructions. Without guardrails, agents can retry indefinitely. Add rules like this to your soul/constitution document:
# In your agent's instruction document: "If a task fails 3 times in a row, stop and report back. Do not retry automatically more than 3 times. If you are uncertain whether to continue, ask rather than guess."
✓
Check for Anthropic rate limit errors (HTTP 429). Sudden silence often means you've hit a rate limit, not a logic error.
openclaw gateway stop openclaw gateway run --verbose # Look for: 429, "rate limit", "overloaded" in the output # If you see 429s: you're either on a low tier or in a burst loop # Fix: add retry delays, or upgrade to Tier 2 on Anthropic console # Ctrl+C to stop, then: openclaw gateway start
